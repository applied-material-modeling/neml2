    "# How compilation works\n",
    "\n",
    "The [previous tutorial](tutorials-models-compiled) showed *how* to compile a model\n",

    "## A model with a shared subexpression\n",
    "\n",
    "To make the optimization visible we need an example where the same quantity is\n",
    "computed more than once. Three first-order rates sharing a denominator do the job:\n",
    "\n",
    "$$\n",
    "  \\dot{a} = \\frac{a - a_n}{t - t_n}, \\qquad\n",
    "  \\dot{b} = \\frac{b - b_n}{t - t_n}, \\qquad\n",
    "  \\dot{c} = \\frac{c - c_n}{t - t_n},\n",
    "$$\n",
    "\n",
    "where the subscript $n$ is the previous time step. Each equation is a\n",
    "`ScalarVariableRate`; composing the three gives a model that takes\n",
    "$a, a_n, b, b_n, c, c_n, t, t_n$ and returns $\\dot a, \\dot b, \\dot c$. The\n",
    "denominator $t - t_n$ is **shared** across all three — a redundancy the compiler\n",
    "should be able to remove."
   ]
  },

    "  []\n",
    "  [eq3]\n",

    "declared input (a batch of 2 here, so the batch axis is a real dimension):"

    "from torch.export import export\n",
    "\n",
    "# One example input per declared input, in `input_spec` order.\n",

    "exported = export(model, example_inputs, strict=False)\n",
    "\n",
    "# Print just the captured forward body (the tensor operations).\n",
    "graph_src = exported.graph_module.print_readable(print_output=False)\n",
    "for line in graph_src.splitlines():\n",

    "print(\"...\")\n",
    "seen = set()\n",
    "for line in src.splitlines():\n",
    "    s = line.strip()\n",
    "    if s.startswith(\"auto tmp\") and (\" - tmp\" in s or \" / tmp\" in s):\n",

    "        seen.add(s)\n",

  {
   "cell_type": "markdown",
   "id": "readkernel",
   "metadata": {},
   "source": [
    "Three things happened here, all visible in those seven lines:\n",
    "\n",
    "- **Common-subexpression elimination.** `tmp5 = tmp3 - tmp4` is the denominator\n",
    "  $t - t_n$, computed **once** and reused by all three divisions (`tmp6`,\n",
    "  `tmp10`, `tmp14`). The threefold redundancy from the FX graph is gone.\n",
    "- **Fusion.** All three rates live in a single function (`cpp_fused_div_sub_0`),\n",
    "  one loop over the batch — not three separate kernels writing intermediates to\n",
    "  memory and reading them back.\n",
    "- **Vectorization.** The arithmetic is on `at::vec::Vectorized` SIMD registers,\n",
    "  so several batch entries are processed per instruction.\n",
    "\n",
    "This is the same common-subexpression optimization NEML2's older just-in-time\n",
    "engine performed on its trace — but here it lands as real, vectorized, compiled\n",
    "machine code rather than an optimized interpreter graph."

   "source": "## What changes when we compile for the GPU\n\nThis notebook compiled for the CPU, so the kernel above is C++. Targeting CUDA\n(`neml2-compile --device cuda`, which additionally needs `nvcc`) reuses the\n*same* captured FX graph but lowers it differently. A few things worth knowing:\n\n- **Same capture, different codegen.** Stage 1 (the FX graph) is identical —\n  capture is device-agnostic. Only Stage 2 changes: Inductor emits **Triton** GPU\n  kernels instead of C++, then the same optimization passes (CSE, fusion,\n  vectorization) run against the GPU cost model.\n- **Fusion matters more — kernel launch cost.** Every GPU kernel launch carries a\n  fixed overhead (microseconds). Fusing the three rates into one kernel means one\n  launch instead of several; at small-to-moderate batch sizes, where there isn't\n  enough arithmetic to hide it, launch overhead can dominate — so fewer, larger\n  kernels pay off more than on CPU.\n- **Fewer temporaries, less memory traffic.** GPUs are usually\n  memory-bandwidth-bound. Fusion keeps intermediates like $t - t_n$ in registers\n  instead of writing them out to global memory and reading them back; that means\n  fewer temporary buffers and less round-tripping to HBM, which is frequently the\n  real bottleneck rather than the arithmetic itself.\n- **Autotuning.** At compile time Inductor *autotunes* each Triton kernel —\n  block sizes, tiling, the number of warps — by benchmarking candidate\n  configurations for the target shape and keeping the fastest. It's a one-time\n  compile cost baked into the artifact. CUDA-graph capture (replaying a whole\n  launch sequence as one unit) is also available for launch-bound graphs.\n- **Same artifact contract.** The result is still a `.pt2`, just holding a CUDA\n  `.so`; `--device cpu cuda` builds both under sibling `<device>/<dtype>/` leaves,\n  and the load-time `device`/`dtype` pick which one (see\n  [](tutorials-models-compiled)). Device and dtype are pinned at compile time."
